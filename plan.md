# CVE Backporting Engine 技术优化计划

目标：不改变现有核心算法主干，优先解决“用户看不懂为什么这么判”的问题，让工具稳定回答三件事：

- [ ] 哪些补丁可以直接回移
- [ ] 哪些补丁需要考虑或不需要考虑关联补丁
- [ ] 哪些变更虽然小，但可能有较大影响

---

## 当前阶段的核心判断

- [ ] 当前最大问题不是算法分支不够多，而是“结论和证据之间没有稳定映射”
- [ ] 低级别准确率不高，关键不是 apply 能力，而是缺少正向准入条件和负向否决条件
- [ ] 最容易被低估的高风险点，是锁、生命周期、状态机、结构体字段、错误路径
- [ ] “单行修改”必须看控制流、同步语义、状态迁移，不能按行数判低风险

## 总体原则

- [ ] 不引入面向用户的复杂评分模型，避免“分数看起来专业，但用户不知道为什么”
- [ ] 结论必须由过程和证据支撑，而不是由黑盒规则堆叠得出
- [ ] 用户首先看到的是“分析过程”和“结论依据”，不是抽象等级或分值
- [ ] L0/L1 的目标是高精度和可直接执行，不是简单把更多样本压到低级别
- [ ] 对高风险场景，宁可明确指出“风险点在哪”，也不要只给一个模糊级别

---

## 北极星目标

### 目标 1：让用户看懂分析过程
- [x] `analyze / validate / batch-validate` 都输出统一的分析过程
- [ ] 每个结论都能回答：
  - [ ] 看了哪些东西
  - [ ] 发现了哪些证据
  - [ ] 为什么得出这个结论
  - [ ] 还需要人工确认什么

### 目标 2：让低级别更可信
- [ ] `L0` 只能是“正向证据充分”的直接回移场景
- [ ] `L1` 只能是“轻微漂移但语义稳定”的轻审查场景
- [ ] 不允许只因“没命中风险规则”就进入低级别

### 目标 3：让高风险识别更准确
- [ ] 锁、生命周期、状态机、结构体字段、错误路径必须有专项分析
- [ ] 单行变更也要检查控制流、同步语义、状态迁移
- [ ] 高风险结论要能定位到具体函数、字段、锁对象、状态点

## P0：把“结论”改成“过程 + 证据 + 结论”

### P0.1 统一分析过程输出
- [x] 为所有主命令统一输出固定过程阶段：
  - [x] fix / intro 搜索过程
  - [x] 关联补丁判断过程
  - [x] dryrun / patch applicability 过程
  - [x] 关键结构与高风险扫描过程
  - [x] 最终结论过程
- [x] 每个阶段都输出“输入了什么、发现了什么、因此怎么判断”的骨架
- [ ] 过程展示要尽量去重，避免 TUI 和 JSON 里反复打印同类信息

### P0.2 统一证据模型
- [x] 为最终结论统一输出三类证据：
  - [x] `direct_backport_evidence`
  - [x] `prerequisite_evidence`
  - [x] `risk_evidence`
- [x] 每条证据都能稳定落到以下信息：
  - [x] 证据来源
  - [x] 命中的对象
  - [x] 命中的原因
  - [x] 对结论的影响
- [ ] 证据来源至少覆盖：
  - [ ] patch diff
  - [ ] dryrun trace
  - [ ] function impact
  - [ ] call chain
  - [ ] struct field / lock / lifecycle / error path

### P0.3 用户面向的结论表达
- [x] 结论层统一收敛成用户能直接理解的三类：
  - [x] `可直接回移`
  - [x] `需要考虑关联补丁`
  - [x] `存在较大影响风险`
- [x] 每类结论都附带“为什么”的结构化骨架
- [ ] 尽量少让用户先理解 `L0-L5`，而是先看结论，再看分级
- [ ] `L0-L5` 保留为内部和批量统计视角，不作为用户第一理解入口

## P1：提升低级别的准确率与可信度

### P1.1 L0 的正向准入条件
- [ ] `L0` 必须满足明确的正向条件，而不是默认放行
- [ ] `L0` 正向准入至少包括：
  - [ ] dryrun 基线为 `strict` 或稳定 `c1`
  - [ ] 补丁修改范围小且边界清晰
  - [ ] 没有关键结构语义变化
  - [ ] 没有函数签名或调用约束变化
  - [ ] 没有强依赖前置补丁
  - [ ] 没有高影响控制流变化
- [ ] `L0` 输出必须明确告诉用户“为什么这个补丁可以直接回移”

### P1.2 L0/L1 的负向否决条件
- [ ] 增加低级别否决项，只要命中就不能轻易留在 `L0/L1`
- [ ] 否决项至少包括：
  - [ ] 锁语义变化
  - [ ] 生命周期变化
  - [ ] 状态机迁移变化
  - [ ] 结构体字段语义变化
  - [ ] 错误路径变化
  - [ ] 返回值语义变化
  - [ ] 条件判断反转或提前返回
- [ ] 否决项命中后，必须把具体风险点直接展示给用户

### P1.3 L1 的“轻微漂移”边界
- [ ] 定义哪些属于轻微上下文漂移，而不是高风险语义变化
- [ ] 优先保留在 `L1` 的场景：
  - [ ] 注释变化
  - [ ] 日志文本变化
  - [ ] 局部变量重命名
  - [ ] 宏名替换但语义等价
  - [ ] 邻近无关代码插入
- [ ] 这些场景要输出“漂移点”而不是直接抬高风险

---

## P2：关键结构、关键语义、高风险场景专项分析

### P2.1 锁与同步语义
- [x] 识别的不只是 `spin_lock/mutex/rcu` 关键词，还要识别：
  - [x] 锁对象本体
  - [x] 加锁/解锁位置变化
  - [x] 锁保护区间变化
  - [x] 锁保护的数据对象
  - [x] 同步顺序变化
- [x] 只要锁相关变更命中，就不能简单按“小改动”处理

### P2.2 生命周期与资源管理
- [x] 重点识别：
  - [x] `alloc/free`
  - [x] `get/put`
  - [x] `refcount_inc/refcount_dec`
  - [x] 初始化/销毁路径
  - [x] 错误回滚路径
- [x] 生命周期变化要回答：
  - [x] 哪个对象的持有关系变了
  - [x] 哪个释放顺序变了
  - [x] 是否可能引入泄漏、悬挂引用、重复释放

### P2.3 状态机与控制流
- [x] 重点识别：
  - [x] 条件分支变化
  - [x] 提前返回/延后返回
  - [x] 错误码变化
  - [x] 状态位或状态字段变化
  - [x] 回调/ops 行为变化
- [x] 对“只改一行”的场景，优先看：
  - [x] 条件符号是否反转
  - [x] 状态判断是否变化
  - [x] 是否改变进入/退出某状态的条件
- [x] 单行修改如果影响控制流或状态迁移，至少按中高风险处理

### P2.4 结构体字段与数据路径
- [x] 对结构体变更输出到字段级别：
  - [x] 新增字段
  - [x] 删除字段
  - [x] 字段类型变化
  - [x] 字段读写位置变化
  - [x] 字段访问路径变化
- [x] 对字段相关变更分析：
  - [x] 哪些函数读它
  - [x] 哪些函数写它
  - [x] 它是否处于锁保护域中
  - [x] 它是否参与状态判断或错误路径

### P2.5 错误路径专项
- [x] 把错误路径从普通控制流里单独拉出来分析
- [x] 重点识别：
  - [x] `goto err`
  - [x] cleanup 逻辑
  - [x] 返回错误码变化
  - [x] 失败后状态恢复变化
- [x] 错误路径变化要明确告诉用户：失败时行为是否变了、清理是否还完整

## P3：关联补丁判断做细，不再只看重叠

### P3.1 关联补丁从“有/无”变成三档
- [x] `required`：不带就可能语义错误或无法生效
- [x] `recommended`：不带也许能过，但风险或维护成本更高
- [x] `independent`：当前补丁在本分支可独立成立

### P3.2 关联补丁判断证据
- [ ] 不只看 hunk overlap，还要看：
  - [ ] function overlap
  - [ ] shared symbol
  - [ ] shared struct field
  - [ ] shared lock domain
  - [ ] shared state transition
- [ ] 输出“为什么认为需要前置补丁”
- [ ] 输出“为什么认为可以不考虑前置补丁”

### P3.3 给用户能直接执行的建议
- [ ] 如果是 `required`，明确指出先带哪些补丁
- [ ] 如果是 `recommended`，明确指出不带会有什么风险
- [ ] 如果是 `independent`，明确指出已经具备哪些前提条件

---

## P4：规则体系改成“准入 + 否决 + 高风险画像”

### P4.1 规则分三类，而不是继续堆阈值
- [x] `准入规则`：用于支持“可直接回移”
- [x] `否决规则`：用于阻止误入低级别
- [x] `高风险画像规则`：用于识别锁、生命周期、状态机、字段、错误路径
- [x] `否决规则` 已继续细化为：
  - [x] `low_level_veto`
  - [x] `direct_backport_veto`

### P4.2 规则输出必须可解释
- [ ] 每个规则命中都要输出：
  - [ ] 命中位置
  - [ ] 命中对象
  - [ ] 命中原因
  - [ ] 导致的结论变化
- [ ] 避免同一事实被多个规则重复打印，减少冗余输出

### P4.3 规则优先级
- [ ] 高风险画像优先于低级别准入
- [ ] 否决规则优先于准入规则
- [ ] 规则之间的提升或否决关系要固定，避免同一样本反复震荡

---

## P5：批量评估从“数量统计”变成“问题统计”

### P5.1 统计的重点不再只是级别数量
- [ ] 统计“为什么进入 L0/L1/L2+”
- [ ] 统计“哪些否决项最常导致低级别失败”
- [ ] 统计“哪些高风险画像最常出现”
- [ ] 统计“关联补丁判断中 required / recommended / independent 的占比”

### P5.2 统计低级别准确率
- [ ] 重点验证：
  - [ ] `L0` 是否真的可以直接回移
  - [ ] `L1` 是否真的只是轻微漂移
  - [ ] 是否有锁/生命周期/状态机变更被误放进低级别
- [ ] 对误判样本输出归因，而不是只给通过率

### P5.3 输出对用户有用的批量结论
- [ ] 当前多少样本可直接处理
- [ ] 当前多少样本需要关联补丁
- [ ] 当前多少样本属于关键结构高风险
- [ ] 当前最常见的风险类型是什么

## 最终目标

- [ ] 用户能直接看懂工具做了哪些分析
- [ ] 用户能直接看懂为什么判成“可直接回移 / 需考虑关联补丁 / 高风险”
